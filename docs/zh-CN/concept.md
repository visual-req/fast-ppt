# Concept

![Outline Detail Workflow](assets/outline-detail-flow.svg)

## 这套系统在解决什么

这套系统不是直接生成一个静态的 `.pptx` 或单张长图，而是生成一套可被前端渲染、可继续回改、可复用的网页 PPT 产物。

最终交付的是：

- `deck.json`
- `slides/*.json`
- `work/assets/*.svg`

前端再基于这些数据完成渲染、预览和跳转。

## 为什么用 JSON 存储

这里选择 JSON，不是因为它“通用”这么简单，而是因为它刚好适合这套生成链路里的三个角色：

- 生成端
- 存储端
- 渲染端

### 原因 1：生成结果天然是结构化数据

PPT 在这个系统里不是一段纯文本，而是：

- deck 元信息
- 页面顺序
- 每页的 `layout_type`
- 标题、卡片、表格、步骤、泳道、图示引用

这些内容本身就是结构化字段，用 JSON 表达最直接，不需要再把结构硬塞回长文本。

### 原因 2：前端读取成本低

`ppt-viewer` 本身就是按：

- `deck.json`
- `slides/*.json`

来读取和渲染页面。

JSON 可以直接被前端解析，不需要额外做文档解析、格式转换或二次抽取。

### 原因 3：便于程序化校验

使用 JSON 后，可以很容易检查：

- 字段是否完整
- `layout_type` 是否合法
- `slide_files` 是否一致
- 引用的 SVG 是否存在
- 某一页是不是缺标题、缺表头、缺步骤

这使得质检可以真正落到程序和规则上，而不是只靠人工目测。

### 原因 4：便于版本化修改

JSON 是文本文件，适合：

- diff
- review
- 局部修改
- 回滚
- 批量替换

这比把整套产物压成一个黑盒格式更适合反复迭代。

## 为什么每个 deck 一套独立文件

这里不是把所有 deck 混在一个总文件里，而是采用：

- `work/ppt/001_项目名/deck.json`
- `work/ppt/001_项目名/slides/*.json`

这样的项目级目录结构。

### 原因 1：一个 deck 是一个完整交付单元

一套 deck 通常包含：

- 自己的标题和元信息
- 自己的页面顺序
- 自己的页面内容
- 自己引用的图示素材

它本身就应该是一个独立单元，而不是散落在一堆共享文件里。

### 原因 2：方便并行生成和回改

不同项目可以各自拥有：

- 自己的 `outline.json`
- 自己的 `deck.json`
- 自己的 `slides/*.json`

这样在改 A 项目时，不会污染 B 项目，也不容易把页面改错到别的 deck。

### 原因 3：便于前端直接加载

服务端只需要知道当前项目目录，就可以聚合：

- `deck.json`
- `slides/*.json`

并返回给前端。

这种组织方式比把所有 deck 放在一个大文件里更简单，也更适合按项目预览。

### 原因 4：便于定位问题

当某一套 PPT 出问题时，可以直接定位到对应目录：

- 先看 `outline.json`
- 再看 `deck.json`
- 再看具体哪一页的 `slides/*.json`

问题定位会非常快，不需要在一份巨大总文件里查找。

### 原因 5：便于后续扩展

后面如果要给某个 deck 增加：

- 主题配置
- 特定资源
- 定制布局
- 单独导出结果

按项目单独存放会更容易扩展，而不会牵一发而动全身。

## 四种 PPT 类型

生成前必须先确定 `deck.type`，目前支持四类：

- `课件类`
- `方案类`
- `汇报类`
- `演示类`

这样分的原因不是命名不同，而是它们的结构目标不同：

- `课件类`：强调讲授顺序、知识拆解、练习和带走什么
- `方案类`：强调问题、方案、架构、路径和风险
- `汇报类`：强调结论、指标、进展、对比和下一步
- `演示类`：强调叙事节奏、重点页面和现场展示效果

类型先确定，后续页面节奏、图页比例、layout 选择和文案密度才会稳定。

## 为什么先生成 outline，再生成 detail

核心原因是：`outline` 负责定结构，`detail` 负责落内容，两者解决的问题不同。

### `outline` 先解决什么

- 这套 PPT 属于哪一类
- 章节如何组织
- 页面顺序是否合理
- 每页的页面意图是什么
- 每页应该选什么 layout

如果这些问题不先定下来，后面直接生成详细页，通常会出现：

- 页序混乱
- 重要图页缺失
- 对比页误用步骤图
- 工作流页误用表格
- 三层结构又退回 bullets

### `detail` 再解决什么

在 `outline` 确认后，`detail` 再负责：

- 写出 `deck.json`
- 写出 `slides/*.json`
- 生成或引用 `work/assets/*.svg`
- 把页面补足到可预览、可质检的程度

也就是说，`outline` 是“先决定做什么”，`detail` 是“再把它做出来”。

## 为什么生成后还要渲染

因为 JSON 只是中间表达，不是最终阅读体验。

渲染这一步的作用是把：

- 标题
- 卡片
- 表格
- 步骤图
- 泳道图
- SVG 图示

统一变成真实页面，方便在浏览器里检查：

- 页面有没有超出屏幕
- layout 选得对不对
- 图形是否真的显示
- 标题、内容和视觉层次是否合理

所以流程不是“生成结束就完事”，而是“生成后必须进入前端预览和质检”。

## 为什么选 SVG 生成图形

优先选 SVG，不是为了炫技，而是因为它更适合这类工程化 PPT 系统。

### 原因 1：可精确控制

SVG 可以直接控制：

- 位置
- 尺寸
- 字体
- 颜色
- 箭头
- 连线
- 层级

这对结构图、关系图、三层图、流程图尤其重要。

### 原因 2：便于版本化修改

SVG 是文本文件，修改成本低，适合：

- 改中文
- 改布局
- 改颜色
- 改卡片尺寸
- 改对齐方式

这比回头重新出一张位图更适合频繁迭代。

### 原因 3：缩放稳定

SVG 在不同屏幕尺寸下仍然保持清晰，不容易像位图一样在放大后变糊。

## 为什么支持 ECharts

支持 ECharts 的原因，是因为不是所有图都适合手写 SVG。

当页面需要：

- 柱状图
- 折线图
- 饼图
- 雷达图
- 面积图

这类标准数据图表时，ECharts 更合适。

原因包括：

- 图表类型成熟
- 数据驱动更直接
- 轴、图例、标签这些基础能力现成
- 前端渲染成本低于为每种数据图单独重写 SVG 模板

所以这里的原则不是“只用 SVG”或“只用 ECharts”，而是：

- 结构图、关系图、流程图优先 SVG
- 标准数据图优先 ECharts

## 为什么不用高清图

这里不优先用高清位图，主要有四个原因：

### 原因 1：位图不方便改

如果标题、箭头、中文文案错了，位图很难局部修改，通常要整张重做。

### 原因 2：工程化复用差

位图更像一次性交付物，不适合在 JSON + 前端渲染这条链路里持续演化。

### 原因 3：清晰度和适配性不稳定

浏览器全屏、缩放、不同分辨率下，位图更容易模糊或裁切不理想。

### 原因 4：难以结构化表达

这套系统强调“图形是结构表达的一部分”，而不是只放一张好看的插图。

SVG 和 ECharts 都是可结构化、可控制、可回改的表达方式，而高清图更多只是最终像素结果。

## 总结

这套系统的核心选择可以概括成一句话：

- 先按四种类型确定结构
- 再先做 `outline`
- 再做 `detail`
- 再进入前端渲染和质检
- 图形优先用 SVG，标准数据图支持 ECharts
- 不优先依赖难以回改的高清位图
