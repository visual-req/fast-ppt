## 适用场景

适用于培训课件、内部教学、知识宣讲、方法论讲解、入门教程、专题课程等内容。

## 基本设计思路

课件类 PPT 的核心目标不是“说服”而是“教会”。整体设计应围绕“循序渐进、降低理解门槛、促进吸收与互动”展开。

默认采用“详尽课件模式”。这意味着：

- 默认优先保证教学完整性，而不是页数最少
- 默认优先让讲师“有内容可讲”，而不是只给出提纲式页面
- 默认优先把知识讲透、讲顺，再考虑压缩
- 只要主题稍复杂，就应主动展开成多页，而不是等待用户逐轮追加强化要求

视觉表达上要尽量减少 `title_bullets` 的使用。除非是目录页、总结页或确实只有 3-4 个必须并列呈现的要点，否则优先使用更有表现力的布局，例如 `two_column`、`top_bottom`、`steps`、`four_grid`、`comparison_table`、`before_after`、`mind_map`。

课件类常见错误是“页数太少、每页塞太多”。当前 skill 在生成课件时，必须显式避免“为了压缩页数而压缩教学过程”。宁可多 3-8 页，也不要把多个知识动作挤在同一页里。

另一个常见错误是“虽然换了布局，但每个块里仍然塞满 bullets”。这会让页面看起来仍像 Word 列表，而不是课件。课件页要优先使用：

- 短标题 + 短句说明
- 卡片式分块
- 表格对比
- 上下结构说明
- 左右结构中的图文对应

只有在确实需要逐条列举时才使用 bullets，而且单块内尽量不超过 2-3 条。

每一个“节”应尽量由以下四部分构成：

1. 概念介绍
2. 详细解析（视情况决定是否展开）
3. 案例介绍
4. 课堂练习或课堂讨论

如果某一节内容较简单，可弱化“详细解析”；但“概念介绍”和“案例/练习或讨论”原则上应保留。

## 节结构设计

建议一个章节按“节”组织，每节推荐页面顺序如下：

1. 节标题页：说明本节主题、学习目标、预期收获
2. 概念介绍：定义、背景、关键术语、适用边界
3. 详细解析：结构拆解、步骤说明、原理说明、注意事项
4. 案例介绍：用真实或仿真案例说明“怎么用、为什么这样做”
5. 课堂练习或课堂讨论：给出题目、讨论点、判断题、开放问题
6. 小结页：回顾本节核心要点

页数原则：

- 课件优先保证教学节奏，不优先压缩页数
- 一个复杂概念可以拆成 2-4 页，不必强行压成 1 页
- 一个复杂系统可以拆成“总览页 + 部件页 + 案例页 + 练习页”
- 如果某页同时承担“定义 + 对比 + 例子 + 注意事项”，通常说明这页应该继续拆开

课堂练习或课堂讨论页默认优先使用 `two_column`：

- 左侧：题目、讨论要求、回答提示、讲师点评方向
- 右侧：配图、示意图、流程截图、结构图或案例图

如果暂时没有图片，右侧也要预留图片位，并说明建议从 `work/assets/` 引用哪类素材。

## 详尽课件生成模式

为了让 prompt 具备“重复生成详尽课件”的能力，生成时必须遵守以下默认方法，而不是依赖单个项目临时发挥。

### 1. 页数预算规则

- 课件默认不走“精简汇报页数”逻辑
- 若输入没有明确页数限制，默认按“中高细节”规划
- 一个完整课程通常至少包含：封面、目录、多个章节、练习/讨论、小结、总结
- 一个“章”通常不少于 `6-10` 页
- 一个复杂核心概念通常不少于 `2-4` 页
- 一个方法/框架类主题通常不少于 `8-12` 页
- 一个系统构成类主题通常不少于 `8-15` 页
- 如果出现多个并列核心概念，页数要按概念数线性增加，而不是压在同几页里

### 2. 主题识别后套用模板

生成课件时，先判断每一章更接近哪类主题，然后套用对应模板。

#### A. 多概念并列主题模板

适用于：提示词工程 / 上下文工程 / Harness 工程，或其他 3 个以上相关并列概念。

推荐顺序：

1. 本章引入页
2. 概念 A 详细介绍
3. 概念 A 应用场景 / 示例
4. 概念 B 详细介绍
5. 概念 B 应用场景 / 示例
6. 概念 C 详细介绍
7. 概念 C 应用场景 / 示例
8. 演进逻辑页
9. 注意事项 / 方法要领页
10. 统一对比页
11. 课堂讨论或练习
12. 小结

#### B. 方法 / 框架 / 能力单元主题模板

适用于：Skill、Harness、Agent 组件、评估体系、工作流框架等。

推荐顺序：

1. 它是干什么用的
2. 在什么情况下需要它
3. 设计思路
4. 设计流程 / 组织方式
5. 目录结构
6. 必要文档
7. 质量保证
8. 优化方法
9. 项目中的真实案例
10. 课堂练习
11. 小结

#### C. 系统构成主题模板

适用于：Harness、Agent Runtime、评估系统、平台架构等。

推荐顺序：

1. 总览页
2. 总体关系图 / 大图页
3. 各核心部件逐页细讲
4. 边界对比页
5. 重点部件深讲页
6. 项目中的规则设计页
7. 课堂练习
8. 小结

#### D. Harness / 执行框架主题增强模板

适用于：Harness、Agent 执行框架、运行时编排、任务闭环系统等。

如果章节主题明确是 Harness 或类似“执行框架”，则在系统构成模板基础上，必须进一步补齐：

1. 它是干什么用的
2. 在什么情况下需要 Harness，而不只是提示词或上下文
3. Harness 的设计思路
4. Harness 的运行组织方式
5. Harness 的质量保证方式
6. Harness 的优化方法
7. 总览图 / 关系图
8. 组件逐页细讲
9. 边界对比
10. 重点组件深讲（如 Evaluator）
11. 项目中的规则设计 / 实例
12. 练习
13. 小结

其中第 3-6 项不能省略，因为 Harness 类主题如果只讲组件，而不讲“如何设计、如何组织、如何保障、如何演进”，就仍然是不完整的。

### 3. 页面展开判定规则

如果出现以下情况，必须继续拆页，而不是保留在一页里：

- 同一页同时承担“定义 + 例子 + 对比 + 注意事项”
- 同一页出现 3 个以上彼此独立的知识动作
- 同一页既要讲概念，又要讲方法，又要讲案例
- 同一页里一个块出现超过 3 条 bullets
- 一章里涉及多个并列主题，却没有为每个主题提供独立讲解页

### 4. 内容充分性默认要求

除非输入非常简单，否则每章默认至少补齐：

- 概念页
- 详细解释页
- 应用场景或案例页
- 练习 / 讨论页
- 小结页

如果是工程方法论课件，还要默认补齐：

- 设计思路
- 质量保证
- 优化方法
- 目录结构
- 必要文档

### 5. 配图默认要求

- 课件中的说明页、练习页、系统总览页，默认应考虑配图
- 如果没有现成素材，也要显式预留 `work/assets/` 的引用位
- 对结构图、关系图、总览图，优先考虑 `svg_full`
- 对练习页，默认左题右图
- 对框架讲解页，尽量做到“文字解释 + 结构图”配套出现
- 对“发展历程、方法演进、框架关系、组件协同、质量闭环”类内容，优先直接生成真实 SVG 资产，而不是只写“建议配图”
- 每一章至少要有 `1` 张真正的图页或图表页；如果整章只出现文字页、表格页和“配图建议”，则视为视觉化不达标
- 练习 / 讨论页右侧若使用 `image`，详细阶段必须输出真实素材文件；不能只留下路径占位和说明文字
- 若输入没有现成截图，也应自行生成结构示意图、流程图、关系图或白板式 SVG 图，用于降低学员理解门槛
- 对于“多概念并列”主题，至少要有 `1` 张演进总览图；对于“方法 / 框架”主题，至少要有 `1` 张结构图；对于“系统构成”主题，至少要有 `1` 张总览关系图

### 6. 表达优化默认要求

生成课件时，默认还要把页面文案从“文档表达”进一步改成“课堂表达”：

- 目录页不要只是罗列章标题，应写成更像课程导入的表达，让学员一眼知道“这节课会先讲什么、再讲什么、最后带走什么”
- 概念页不要只写抽象定义，应优先回答“它先解决什么问题”，再解释“它是什么”
- 演进页不要写成静态历史摘要，应写成“为什么前一代不够，所以才走到下一代”的推进关系
- 组件页不要只讲名词解释，应优先讲“它的作用是什么、为什么离不开它、在本题里对应什么”
- 练习页不要像需求清单，应写成更像老师发问的句式，让学员知道“要讨论什么、先从哪里想”
- 小结页不要重复章节标题，应提炼成“这节课真正要记住的 3-5 句话”
- 若一页文案读起来像说明书、设计文档或目录提纲，就说明还需要继续改写

## `/fppt:outline` 生成要求

- `/fppt:outline` 必须先写出 `work/ppt/001_项目名/outline.json`，用于确认章节结构、页序、布局选择和每页教学目的
- `outline.json` 是进入 `/fppt:detail` 之前的必要检查点；未生成或未确认时，不允许直接生成详细页面内容
- 先识别课程主题、受众、时长、章节数和每节学习目标
- 大纲必须按“章-节”组织，而不是按汇报逻辑组织
- 每节至少规划 4 类内容：概念介绍、案例介绍、练习/讨论、小结
- 详细解析部分至少要有 1 页；只要涉及结构拆解、原理说明、步骤说明、注意事项中的两项及以上，就应主动拆成多页
- 在 `slides/*.json` 中标明每页教学目的，例如“定义概念”“解释原理”“展示案例”“课堂互动”
- 对“发展历程、方法演进、阶段差异”类内容，优先使用 `comparison_table`、`before_after`、`two_column` 或多页对比结构，把各代差异明确讲出来
- 如果主题中存在 3 个以上并列且彼此相关的核心概念，例如“A 工程 / B 工程 / C 工程”，不要一上来就做总对比；应先为每个概念各给至少 1 页详细介绍，再为每个概念补 1 页应用场景或案例示例，最后再做统一对比页
- 如果主题中存在“方法 / 框架 / 模块 / 能力单元”本身，例如 Skill、Harness、Agent 组件、评估体系，不要只讲定义；应主动展开联想，至少覆盖：`它是干什么用的`、`为什么要这样设计`、`设计思路`、`质量保证`、`优化方法`、`目录结构`、`必要文档`、`项目中的实例`
- 对“系统构成”类内容，不能一页粗略带过；当对象由多个部件组成时，必须拆成总览 + 分部讲解
- 对 Harness 这类概念，至少覆盖 `context`、`tools`、`environment`、`memory`、`planner`、`evaluator`
- 当主题本身就是“系统工程/框架工程/平台构成”时，必须采用“1 页总览 + 每个核心组成至少 1 页细讲 + 1 页整体串联”的结构
- 对每个组成部分，至少讲清 5 个维度：`作用是什么`、`输入输出是什么`、`设计要点是什么`、`常见问题是什么`、`在当前项目中的例子是什么`
- 对提示词工程类内容，要补充注意事项与实操要领，例如“如何提高提示词精准度、如何减少歧义、如何控制输出格式”
- 对复杂页，优先拆成多页或多块短句，不要在一个布局里继续堆很多 bullets
- 默认采用“详尽课件模式”，主动补齐教学链路，不要等用户多轮追加强化
- 先做“章节类型识别”，再套用对应章节模板，而不是全课统一套一个简单骨架
- 如果一个章节属于方法/框架类主题，默认补齐“用途、设计、组织、保障、优化、实例、练习、小结”
- 如果一个章节属于多概念并列主题，默认补齐“逐个讲解 + 逐个场景 + 演进 + 对比 + 练习”
- 如果一个章节属于系统构成类主题，默认补齐“总览 + 组件逐页细讲 + 边界对比 + 重点部件深讲”
- 如果一个章节属于 Harness / 执行框架类主题，默认补齐“用途 + 需要场景 + 设计思路 + 运行组织 + 质量保证 + 优化方法 + 组件展开”
- 若输入未限制页数，不要主动压缩页数；应优先保证内容完整和教学节奏
- `deck.json + slides/*.json` 需要体现这种展开能力，不能只停留在章标题和少量页名
- 目录页、概念页、组件页、小结页在大纲阶段就应写成“可讲”的页面意图，不要只写文档式页名

## `/fppt:detail` 细化要求

- `/fppt:detail` 的前提是项目目录内已存在并确认 `outline.json`
- `deck.json + slides/*.json` 的页序、章节组织和页面意图，必须与已确认的 `outline.json` 对齐；如需改结构，应先回到大纲阶段
- 概念介绍页：补充定义、关键词、边界、易混点
- 详细解析页：补充分步骤说明、因果关系、原理解释、常见误区
- 概念应用页：补充该概念适用于什么任务、不适用于什么任务、至少 1 个贴近主题的场景示例
- 方法框架页：补充该方法或框架的用途、设计原则、质量保证方式、优化思路、目录组织和文档约定
- 系统组成细讲页：补充部件职责、上下游关系、输入输出、设计注意事项、项目实例
- Harness 框架页：补充为什么需要它、如何组织任务流、如何做质量控制、如何持续迭代
- 案例介绍页：补充背景、做法、结果、启示
- 练习/讨论页：补充题目、讨论引导、回答方向、教师提示
- 小结页：提炼 3-5 条本节必须记住的结论
- 目录页：补充课程推进逻辑，不要只罗列章节名
- 演进页：补充“为什么上一代不够、下一代多解决了什么”
- 组件细讲页：补充“作用、输入输出、为什么关键、本题例子、注意事项”
- 配图页：补充真实图片或 SVG 素材、图片用途、与左侧内容的对应关系；若未提供素材，则主动生成 `work/assets/` 下的示意图，不要只写建议
- 细化时不允许把新增信息继续粗暴堆成 bullets，应优先改写成短句说明、卡片文案、对比表述和图文配对
- 如果发现某页细化后信息明显过载，应回退到拆页，而不是把内容硬塞进一页

`/fppt:detail` 默认细化动作：

1. 把过于抽象的标题改成更可讲的页面表达
2. 把“只有结论”的页面补成“结论 + 解释”结构
3. 把“只有定义”的页面补成“定义 + 适用场景 / 案例”
4. 把方法类页面补成“用途 + 设计 + 质量 + 优化”
5. 把系统类页面补成“总览 + 运行逻辑 + 组件解释 + 规则示例”
6. 把课堂练习页补成“题目 + 讨论要求 + 讲师点评方向 + 配图建议”
7. 把过于像文档提纲的 bullets，改成更短的教学句式或卡片内容
8. 把抽象定义页改成“先讲它解决什么问题，再讲它是什么”
9. 把组件页改成“作用 + 输入输出 + 本题例子 + 注意事项”结构
10. 把小结页改成更强结论式表达，而不是重复目录标题
11. 把“建议配图”页落实成真实图页、SVG 图页或可渲染的图表页
12. 把纯文字对比页优先改成演进图、关系图、结构图或图文双栏

`/fppt:detail` 细化后的页面，应尽量达到以下状态：

- 讲师拿到页面后，基本知道这页要讲什么、怎么讲
- 学员看到页面后，知道这页的知识重点和例子是什么
- 页面本身已经显现出“教学节奏”，而不是仍停留在目录式提纲

## 质检环节

课件类生成时，必须默认插入质检环节，而不是生成完就直接结束。

质检分为两轮：

### A. `outline.json` 质检

在 `/fppt:outline` 产出 `outline.json` 后，必须先检查：

- 章节是否按“章-节”组织，而不是散页堆叠
- 每节是否至少具备“概念介绍 / 案例介绍 / 练习或讨论 / 小结”中的三项
- 复杂概念、方法框架、系统构成、多概念并列主题是否已经主动拆页
- 页面标题是否已经写成“可讲”的表达，而不是文档式标题
- 目录页、概念页、组件页、小结页是否具备明确教学意图
- 练习页是否已经规划成左题右图或至少预留图片位
- 是否存在“明显页数过少，教学链路不完整”的情况

若以上任一项不满足，必须先回改 `outline.json`，不能直接进入 `/fppt:detail`。

### B. `deck.json + slides/*.json` 质检

在 `/fppt:detail` 生成详细页后，必须再检查：

- 页序、章节组织和页面意图是否仍与已确认的 `outline.json` 对齐
- 是否仍有页面像目录提纲、说明书或设计文档，而不像课堂页面
- 是否仍有长句过多、整块 bullets 过多、换了布局但还是列表堆积的问题
- 概念页是否先讲“解决什么问题”，再讲“它是什么”
- 演进页是否讲清“为什么上一代不够、下一代多解决了什么”
- 组件页是否讲清“作用、输入输出、为什么关键、本题例子、注意事项”
- 小结页是否已经提炼成可记忆的结论，而不是重复章节标题
- 练习页是否像老师发问，而不是把要求直接写成任务清单
- 系统页、框架页、练习页是否已补足配图建议或结构图位置
- 系统页、框架页、练习页是否已经有真实素材或真实结构图，而不只是“建议配图”
- 每章是否至少出现一张真正的图页 / 图表页 / 结构图页

若以上任一项不满足，必须先回改对应页面，不允许把“不够像课件”的结果直接作为最终交付。

### C. 质检输出要求

每次质检都应按固定模板输出检查结论：

1. `通过项`
2. `未通过项`
3. `修改动作`
4. `阶段判定`

推荐格式：

```md
质检结论

- 通过项：
  - ...
- 未通过项：
  - ...
- 修改动作：
  - ...
- 阶段判定：允许进入下一阶段 / 必须回改后再检查
```

其中 `阶段判定` 不能省略；如果没有明确写出阶段判定，则默认视为当前阶段未通过。

`/fppt:outline` 质检示例：

```md
质检结论

- 通过项：
  - 已按“章-节”组织课程结构
  - Skill、Harness 两个核心章节都已拆成独立部分
  - 练习页已规划为左题右图结构
- 未通过项：
  - “提示词工程、上下文工程、Harness 工程”目前只有总对比页，缺少逐个讲解页
  - Harness 章节只有总览，没有把 `context`、`tools`、`environment`、`memory`、`planner`、`evaluator` 拆开
- 修改动作：
  - 为三个核心概念各补“概念介绍页 + 场景页”
  - 为 Harness 增加总览页、组件逐页细讲页和整体串联页
- 阶段判定：必须回改后再检查
```

`/fppt:detail` 质检示例：

```md
质检结论

- 通过项：
  - 页序与已确认的 `outline.json` 保持一致
  - 组件页已经补齐作用、输入输出和本题例子
  - 小结页已改成结论式表达
- 未通过项：
  - 两张概念页仍然先给定义，没有先讲“解决什么问题”
  - 一张演进页仍像文档摘要，没有讲清“上一代为什么不够”
  - 一张练习页右侧仍缺少配图建议
- 修改动作：
  - 重写概念页开头，先补问题导入，再给定义
  - 把演进页改成“上一代限制 -> 下一代新增能力”的推进表达
  - 为练习页补充 `work/assets/` 图片占位和用途说明
- 阶段判定：必须回改后再检查
```

## 生成流程

为了重复生成高质量详尽课件，建议固定遵守以下流程：

1. 读取输入材料，判断课程目标、受众、复杂度
2. 将全课拆成若干章节，并识别每章主题类型
3. 对每章套用合适的章节模板，先定页数下限
4. 先完成 `outline.json`，并执行一轮大纲质检
5. 为每页选择最合适的布局，而不是默认 `title_bullets`
6. 为说明页、练习页、系统页补配图位
7. 生成 `deck.json + slides/*.json` 后，再执行一轮详细页质检
8. 若质检未通过，先回改，再交付

## 自检清单

生成课件后，必须自问：

- 这份课件是否已经足够详细，即使用户不继续追问，也能直接用于讲授？
- 是否存在“本应拆开的知识点被挤在一页里”的情况？
- 是否每个核心概念都有独立讲解页和应用页？
- 是否每个工程方法类主题都覆盖了设计、结构、保障、优化？
- 如果章节主题是 Harness 或执行框架，是否覆盖了用途、场景、设计、组织、保障、优化和组件展开？
- 是否仍然存在“换了布局但本质还是大段 bullets”的页面？
- 是否给练习页、系统页、框架页预留了合适配图？
- 是否已经把关键配图位落实成真实图片或 SVG，而不是只停留在“建议”层？

## 推荐布局

- 概念介绍：`two_column`、`top_bottom`、`four_grid`
- 详细解析：`steps`、`phases`、`top_bottom`、`mind_map`
- 差异对比：`comparison_table`、`before_after`、`two_column`
- 案例介绍：`before_after`、`two_column`、`four_grid`
- 课堂练习/讨论：`two_column`（左题右图）为默认首选，备选 `four_grid`
- 系统构成总览：`mind_map`、`architecture_layered`
- 组成部分细讲：`two_column`、`top_bottom`、`steps`
- 多组件对比：`comparison_table`、`three_column`
- 小结页：`summary`
- 演进对比：`svg_full`、`before_after`、`comparison_table`
- 关系总览：`svg_full`、`mind_map`、`logic_tree`
- 质量闭环：`swimlane_process`、`logic_tree`、`steps`

## 文案风格

- 解释性强于结论性
- 多用“定义、例如、注意、易错点、思考题”等教学语言
- 允许适度多字，但每页仍应只有一个教学重点
- 避免管理汇报式口吻，不要把课件写成方案或汇报
- 页面要“能讲”，不是简单把文档要点堆成列表
- 优先写成老师讲课时会说的话，而不是设计文档、说明书或会议纪要的语气
- 先用一句话讲清问题，再讲定义、方法和注意事项
- 能写成短句就不要写成长句；能拆成两句就不要硬挤成一句
- 目录页要像导入，演进页要像推进，组件页要像拆解，小结页要像收束
- 对比页一定要讲清“前一代解决什么、下一代多解决了什么、代价是什么、注意事项是什么”
- 能用短句说明就不要写长 bullets，能做成卡片就不要做成一整列列表
- 对多概念主题，讲解顺序优先是“分别讲清”而不是“先混在一起做总比较”
- 对 Skill、Harness 这类工程概念，要主动补全“设计、组织、保障、优化”四类信息，而不是停留在名词解释层

## 输出检查

- 是否每一节都包含四部分中的至少三部分
- 是否案例和练习真正服务于知识掌握，而不是形式化占位
- 是否可以让授课者直接按页讲解
- 是否尽量避免了大面积 `title_bullets`
- 课堂练习页是否为左右结构，且右侧有图或明确的配图占位
- 对复杂系统类主题，是否真的做到“总览 + 分部细讲 + 项目实例 + 注意事项”
- 非目录/总结页是否仍有过多 bullets 堆积
- 对多概念主题，是否先给了每个概念的详细介绍页和应用场景页，再进入对比
- 对方法/框架类主题，是否覆盖了用途、设计思路、质量保证、优化方法、目录结构、必要文档和项目实例
