## 适用场景

适用于咨询方案、售前方案、实施方案、项目建议书、建设规划、能力提升方案等内容。

## 基本设计思路

方案类 PPT 的核心目标是“说明问题、提出方案、证明可行、推动决策”。整体结构应强调逻辑闭环和落地路径，而不是单纯展示信息。

默认采用“完整方案模式”。这意味着：

- 默认先保证论证闭环完整，再考虑压缩页数
- 默认让决策者看清“为什么做、做什么、怎么落地、如何控风险”
- 默认把总体方案和关键分项拆开讲，而不是挤在一页里
- 只要主题稍复杂，就应主动展开为“总览 + 分项 + 落地 + 保障”多页结构

推荐主线：

1. 背景与目标
2. 现状与问题
3. 差距与机会
4. 总体方案
5. 分项设计
6. 实施路径
7. 成本收益
8. 风险与保障
9. 下一步建议

## 章节模板

一个完整方案章节通常按以下顺序组织：

1. 章节引入：本章要解决什么问题
2. 现状 / 约束：当前基础、边界、限制条件
3. 方案设计：总体思路、原则、模块、机制
4. 落地路径：阶段划分、关键动作、依赖关系
5. 风险与保障：风险来源、控制方式、组织保障
6. 小结 / 决策提示：本章应形成什么判断

页数原则：

- 总体方案通常不少于 `2-4` 页
- 复杂分项方案通常不少于 `2-3` 页
- 实施路径通常不少于 `1-2` 页
- 风险与保障通常不少于 `1-2` 页
- 如果一页同时承担“问题 + 方案 + 计划 + 风险”，通常说明还需要拆页

## `/fppt:outline` 生成要求

- `/fppt:outline` 必须先写出 `work/ppt/001_项目名/outline.json`，用于确认章节结构、页序、布局选择和每页决策目的
- `outline.json` 是进入 `/fppt:detail` 前的必要检查点；未生成或未确认时，不允许直接生成详细页面内容

### 固定头部页面（必须生成，不可省略）

无论方案主题和页数如何，`outline.json` 必须包含：

1. **封面页**（`cover`）：第一页，`title`/`subtitle`/`date`/`lecturer` 必填
2. **目录页**（`agenda`）：第二页，`bullets` 列出每个章节标题和决策目的
3. **章节封面页**（`section_divider`）：每个 chapter 前必须插入一页章节封面用于转场
4. **致谢页**（`thank_you`）：最后一章最后一页，`title`/`subtitle`/`lecturer`/`contact` 必填，作为 PPT 收尾

示例结构：

- Chapter 0 标题 "开场"：slides = [cover, agenda]
- Chapter 1 标题 "问题与方案闭环"：slides = [section_divider, title_bullets, problem_statement, ...]
- Chapter 2 标题 "实施路径"：slides = [section_divider, roadmap, milestones, ...]
- Chapter N 标题 "收口"：slides = [section_divider, summary, thank_you]

### 页面意图 → layout 映射（方案类增强规则）

以下映射关系在方案类中为强制规则：

- KPI 指标/关键数字/目标量化 → 必须用 `kpi_cards`，禁止用 `title_bullets` 罗列数字
- 端到端机会点/全链路路径/分阶段推进/里程碑演进 → 必须用 `phases`
- 三步/四步/五步方法论 → 用 `steps`
- 执行摘要/总判断/“一个底座 + 多类能力 + 保障机制” → 优先 `title_bullets`，并补 `subtitle` + `cards`
- 系统边界/平台全景/九类模块地图/基础设施总览 → 优先 `nine_grid`
- 自下而上分层/底座-平台-应用/治理-流程-能力-基础设施 → 优先 `architecture_layered`
- 集成链路/追溯链路/样板到复制到规模化 → 优先 `steps`，每步可同时带 `text` + `bullets`
- 如果当前 viewer 没有完全匹配的 layout，或希望版式更自由、画面更强，可直接改用 `svg_full` 实现相同页面意图，不强制为新 layout 先新增 component
- 案例拆解/实践复盘 → 用 `case_study`（默认 `mode: "left_right"`）
- 多维能力评估 → 用 `radar_chart`，必填 `categories`/`values`/`max`

### 通用要求

先抽取项目背景、目标、范围、约束、关键干系人
- 明确“问题是什么、为什么现在要做、方案解决什么”
- 大纲必须覆盖"封面→目录→章节封面→现状→问题→方案→实施→收益→风险→致谢"完整链路
- 在 `outline.json` 中明确每页的决策作用，例如“说明问题严重性”“展示总体方案”“解释实施路径”“说明风险控制”
- 在 `slides/*.json` 中优先体现决策结构，而不是资料罗列
- 页面标题必须写业务主题、阶段结论或对象名称，禁止直接把布局名写进标题；不要出现“九宫格”“四宫格”“步骤图”“流程图”“甘特图”等标题命名
- 对总体方案，默认采用“总览页 + 分项设计页 + 关键机制页”
- 对分专题方案，默认采用“问题界定 + 设计思路 + 方案结构 + 落地方式”
- 对实施内容，默认采用“阶段路径 + 里程碑 / 依赖 + 责任分工”
- 对保障内容，默认采用“风险识别 + 应对机制 + 治理与组织”
- 若一章内存在多个独立子方案，应为每个子方案提供独立讲解页，而不是并在一张总表里带过
- 若输入未限制页数，不要为了“显得短”而跳过收益、风险、保障页

### 页面展开规则（重要）

当一页使用 `phases` / `steps` / `roadmap` 等阶段性布局，且阶段 ≥ 3 个时，必须在总览页后面逐一插入各阶段的详细页：

- 总览页（`phases`）展示 3-5 个阶段概览
- 紧接着插入 N 页详细页，每页对应一个阶段
- 详细页可使用 `title_bullets`、`two_column`、`case_study`、`svg_full`、`four_grid` 等布局，具体根据该阶段的内容特征选择
- 如果详细页本质上是“九类系统总览 / 分层架构 / 线性推进路径 / 执行摘要卡片”，优先考虑 `nine_grid`、`architecture_layered`、`steps`、`title_bullets` 等结构化布局；如果当前 layout 表达力不足，允许直接用 `svg_full` 画成同等结构的信息图
- 当阶段内容涉及系统架构、数据底座、集成拓扑、技术栈全景等需要图示表达的主题时，应优先使用 `svg_full` 并同步在 `work/assets/` 下生成对应 SVG 资产
- 生成 `svg_full` 时，SVG 主体必须主动缩高处理，不要顶满 16:9 画布；图形主体建议控制在画布高度的 `78%-86%`，上下保留 `7%-10%` 安全边距，保证 deck 预览能完整显示
- 阶段命名：如果总览页 slide_no 是 `007`，则详细页编号为 `008_phases_detail`、`009_phases_detail`、`010_phases_detail`
- 大纲 `outline.json` 中不需要为详细页单独列条目，由 `/fppt:detail` 阶段自动展开

示例：007 页用 `phases` 展示 3 个阶段（流程系统建设 / 数据底座搭建 / 数据贯通），则自动生成 008/009/010 三页详细页，分别展开各阶段的具体内容。

### 特殊页面布局建议

- 基础设施/统一底座/平台全景类内容 → 应考虑使用 `nine_grid`（九宫格）布局，9 个平台/能力各占一格
- 执行摘要/高层判断页 → 应考虑使用 `title_bullets + cards`，先给总判断，再给 2-4 个能力或结果卡片
- 三层或多层能力分层 → 应考虑使用 `architecture_layered`，不要退回普通 bullets
- 集成路径/追溯路径/推广路径 → 应考虑使用 `steps`，让“顺序、动作、结果”同时可见
- 新增版式如果只是当前项目里的临时变体，可直接沉淀为 `svg_full` 资产，不要求先抽象成新的 viewer layout component
- 收口/路线图总结 → 可使用 `svg_full` 配合自生成 SVG 时间线展示完整推进路径与启动建议
- 最后一页可使用 `plan_table`（表格）代替 `thank_you`，列出下一步行动项/负责方/时间/产出

## `/fppt:detail` 细化要求
- `/fppt:detail` 的前提是项目目录内已存在并确认 `outline.json`
- `deck.json + slides/*.json` 的页序、章节组织和页面意图，必须与已确认的 `outline.json` 对齐；如需改结构，应先回到大纲阶段

### 固定页面详细生成要求

以下页面的字段必须完备，不允许留空：

**封面页（`cover`）**：
```json
{
  "layout_type": "cover",
  "title": "方案标题（必填）",
  "subtitle": "副标题（必填）",
  "date": "2026-06",
  "lecturer": "讲师名（必填）"
}
```

**目录页（`agenda`）**：
```json
{
  "layout_type": "agenda",
  "title": "目录",
  "bullets": ["第一章 背景与目标 - 说明为什么要做", "第二章 总体方案 - 展示核心设计", "第三章 实施路径 - 说明怎么落地"]
}
```

**章节封面页（`section_divider`）**：
```json
{
  "layout_type": "section_divider",
  "title": "本章标题",
  "subtitle": "本章要回答的核心问题"
}
```

**致谢页（`thank_you`）**：
```json
{
  "layout_type": "thank_you",
  "title": "感谢聆听",
  "subtitle": "以数字化为底座，以 AI 为引擎，共建智能新未来",
  "lecturer": "方案团队",
  "contact": "期待与您深入交流"
}
```
- `title`/`subtitle`/`lecturer` 必填，`contact` 可选
- 致谢页必须放在整个 PPT 的最后一页

### 新类型页面的 JSON 示例

**KPI 页面（`kpi_cards`）**：
```json
{
  "layout_type": "kpi_cards",
  "title": "核心指标总览",
  "cards": [
    { "label": "预期节省成本", "value": 240, "unit": "万元/年", "trend": "同比 +18%" },
    { "label": "交付周期缩短", "value": 45, "unit": "%", "trend": "从 20 天降至 11 天" },
    { "label": "系统可用率", "value": 99.95, "unit": "%", "trend": "高于目标 99.9%" },
    { "label": "覆盖业务线", "value": 6, "unit": "条", "trend": "新增 2 条" }
  ]
}
```

**案例页（`case_study`）**：
```json
{
  "layout_type": "case_study",
  "mode": "left_right",
  "title": "落地案例回顾",
  "image": { "src": "work/assets/case-screenshot.png", "caption": "系统界面截图" },
  "blocks": [
    { "title": "背景", "text": "客户面临库存周转天数过长的问题。" },
    { "title": "做法", "text": "引入需求预测模型和自动补货策略。" },
    { "title": "成效", "text": "周转天数从 45 天降至 28 天。" },
    { "title": "启示", "text": "数据质量是预测模型落地的前提。" }
  ]
}
```
如果没有截图素材，`image.src` 可留空，组件会显示占位区域。

**雷达图（`radar_chart`）**：
```json
{
  "layout_type": "radar_chart",
  "title": "能力成熟度评估",
  "categories": ["需求管理", "架构设计", "质量保障", "运维能力", "安全合规", "数据治理"],
  "values": [3.5, 4.0, 2.5, 3.0, 4.5, 2.0],
  "max": 5
}
```
- `categories`、`values` 必须一一对应，缺一不可
- `max` 是满分值，建议填整数（如 5、10、100）

- 背景与目标页：补充业务背景、建设动因、目标口径
- 现状与问题页：补充现象、影响、根因、证据
- 总体方案页：补充设计原则、方案框架、能力模块、边界说明
- 实施路径页：补充分阶段动作、里程碑、依赖项、责任角色
- 成本收益页：补充投入项、收益项、测算前提、时间范围
- 分项设计页：补充子方案目标、能力点、输入输出、与总体方案的关系
- 组织保障页：补充治理机制、职责分工、决策机制、推进要求
- 细化时若发现信息明显过载，应拆成多页，不要把关键模块继续压成列表
- 对 `svg_full` 页面，除内容校验外还必须做一轮尺寸校验：在 `960x540` 画布中，图形主体不能贴边、不能依赖裁切显示、不能让底部说明被挤出可视区域

## 质检环节

方案类生成时，必须默认插入质检环节，而不是生成完就直接结束。

### A. `outline.json` 质检

在 `/fppt:outline` 产出 `outline.json` 后，必须先检查：

- 是否包含 cover、agenda、section_divider、thank_you 这四个固定页面，且字段完备
- 是否覆盖"背景 / 问题 / 方案 / 落地 / 收益 / 风险"完整链路
- 总体方案、分项方案、实施路径、风险保障是否已经拆开，而不是压在一页
- 每页是否标明明确决策作用，而不是资料罗列
- 是否存在多个子方案却没有独立讲解页的情况
- 是否存在明显页数过少，导致方案闭环不完整的情况
- 标题含"端到端""全链路""阶段""路径"的页面是否确实用了 `phases`/`steps`/`roadmap`
- 标题含"指标""KPI""目标量化"的页面是否确实用了 `kpi_cards`
- 标题含"案例""复盘""落地回顾"的页面是否确实用了 `case_study`

若以上任一项不满足，必须先回改 `outline.json`，不能直接进入 `/fppt:detail`。

### B. `deck.json + slides/*.json` 质检

在 `/fppt:detail` 生成详细页后，必须再检查：

- 页序、章节组织和页面意图是否仍与已确认的 `outline.json` 对齐
- cover 页的 `title`/`subtitle`/`date`/`lecturer` 是否都有值
- radar_chart 的 `categories`/`values`/`max` 是否都填写了，且一一对应
- kpi_cards 的每张卡片是否都有 `label`/`value`/`unit`
- 是否回答了"为什么做、做什么、怎么做、何时做、谁来做、有什么风险"
- 是否仍有页面只写结论，没有论据、路径或约束支撑
- 是否仍有"问题 + 方案 + 计划 + 风险"被硬塞在一页里的情况
- 是否把方案写成培训材料、产品介绍或泛泛而谈的建议清单
- 收益、风险、治理和落地要求是否已经讲清，而不是只停留在方案概念层

若以上任一项不满足，必须先回改对应页面，不允许直接交付。

### C. 质检输出要求

每次质检都应按固定模板输出检查结论：

1. `通过项`
2. `未通过项`
3. `修改动作`
4. `阶段判定`

推荐格式：

```md
质检结论

- 通过项：
  - ...
- 未通过项：
  - ...
- 修改动作：
  - ...
- 阶段判定：允许进入下一阶段 / 必须回改后再检查
```

其中 `阶段判定` 不能省略；如果没有明确写出阶段判定，则默认视为当前阶段未通过。

质检示例：

```md
质检结论

- 通过项：
  - 已覆盖背景、问题、总体方案、实施路径四个核心部分
  - 总体方案已经从现状分析中独立出来
- 未通过项：
  - 风险与保障仍未单独成页，当前闭环不完整
  - 两个子方案仍放在同一页里，缺少各自的目标、能力点和落地方式
- 修改动作：
  - 补一页“风险识别 + 应对机制”，再补一页“治理与组织保障”
  - 将两个子方案拆成各自独立页面，分别补齐目标、能力点、输入输出和实施关系
- 阶段判定：必须回改后再检查
```

## 推荐布局

- 封面/目录/章节封面/致谢：`cover`、`agenda`、`section_divider`、`thank_you`（固定页面）
- 背景/目标/执行摘要：`title_bullets`、`two_column`
- KPI 指标/核心量化：`kpi_cards`
- 现状/问题：`problem_statement`、`matrix_2x2`
- 能力评估/多维度：`radar_chart`
- 总体方案/内容分层：`architecture_layered`、`pyramid`、`logic_tree`
- 方案对比：`comparison_table`、`three_column`
- 阶段推进/端到端路径：`phases`、`steps`、`nine_grid`
- 实施路径：`roadmap`、`milestones`、`plan_table`、`gantt_chart`
- 案例/复盘：`case_study`
- 收益与风险：`cost_benefit`、`risk_register`、`raci`、`impact_effort`
- 流程泳道：`swimlane_process`
- 输入→处理→输出链路、平台处理链路、阶段步骤 + 成果卡：`process_flow`

## 可视化补充要求

- 方案类默认不能只靠文字论证，至少要显式产出以下视觉页面：
  1. 封面 + 目录 + 章节封面 + 致谢（固定）
  2. 总体架构图 / 方案结构图
  3. KPI 仪表盘（如有核心指标）
  4. 阶段图 / 实施路径图
  5. 收益或风险页
- 如方案中有核心量化指标，必须用 `kpi_cards` 展示，不能散落在 bullets 里
- 如方案中有能力评估/成熟度评估，必须用 `radar_chart` 展示，且 `categories`/`values`/`max` 必填
- 如方案中有实践复盘/落地案例，必须用 `case_study` 展示
- 总体方案页优先使用 `architecture_layered`、`logic_tree`、`svg_full`，不要只把模块名称堆成 bullets
- 实施路径页优先使用 `roadmap`、`milestones`、`gantt_chart`、`phases`、`process_flow`、`swimlane_process`，不要只写阶段说明段落
- 收益、优先级、资源投入和风险治理，优先使用 `cost_benefit`、`impact_effort`、`risk_register`、`comparison_table`
- 若没有现成示意图，也应在 `work/assets/` 下主动生成 SVG 结构图或流程图，不能把"建议放架构图"作为最终交付

## 文案风格

- 结论先行，但必须有论据支撑
- 关注“可落地、可执行、可评估”
- 多用“建议、应当、优先级、阶段、交付物、依赖、收益、风险”等方案语言
- 避免写成培训材料或产品宣传页

## 输出检查

- 是否回答了“为什么做、做什么、怎么做、何时做、谁来做、有什么风险”
- 是否具备从立项讨论进入执行准备的基本完整度
- 是否每个核心建议都能在后文找到支撑
