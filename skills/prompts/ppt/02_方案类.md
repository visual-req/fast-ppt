## 适用场景

适用于咨询方案、售前方案、实施方案、项目建议书、建设规划、能力提升方案等内容。

## 基本设计思路

方案类 PPT 的核心目标是“说明问题、提出方案、证明可行、推动决策”。整体结构应强调逻辑闭环和落地路径，而不是单纯展示信息。

默认采用“完整方案模式”。这意味着：

- 默认先保证论证闭环完整，再考虑压缩页数
- 默认让决策者看清“为什么做、做什么、怎么落地、如何控风险”
- 默认把总体方案和关键分项拆开讲，而不是挤在一页里
- 只要主题稍复杂，就应主动展开为“总览 + 分项 + 落地 + 保障”多页结构

推荐主线：

1. 背景与目标
2. 现状与问题
3. 差距与机会
4. 总体方案
5. 分项设计
6. 实施路径
7. 成本收益
8. 风险与保障
9. 下一步建议

## 章节模板

一个完整方案章节通常按以下顺序组织：

1. 章节引入：本章要解决什么问题
2. 现状 / 约束：当前基础、边界、限制条件
3. 方案设计：总体思路、原则、模块、机制
4. 落地路径：阶段划分、关键动作、依赖关系
5. 风险与保障：风险来源、控制方式、组织保障
6. 小结 / 决策提示：本章应形成什么判断

页数原则：

- 总体方案通常不少于 `2-4` 页
- 复杂分项方案通常不少于 `2-3` 页
- 实施路径通常不少于 `1-2` 页
- 风险与保障通常不少于 `1-2` 页
- 如果一页同时承担“问题 + 方案 + 计划 + 风险”，通常说明还需要拆页

## `/fppt:outline` 生成要求

- `/fppt:outline` 必须先写出 `work/ppt/001_项目名/outline.json`，用于确认章节结构、页序、布局选择和每页决策目的
- `outline.json` 是进入 `/fppt:detail` 前的必要检查点；未生成或未确认时，不允许直接生成详细页面内容

- 先抽取项目背景、目标、范围、约束、关键干系人
- 明确“问题是什么、为什么现在要做、方案解决什么”
- 大纲必须覆盖“现状-问题-方案-实施-收益-风险”完整链路
- 在 `outline.json` 中明确每页的决策作用，例如“说明问题严重性”“展示总体方案”“解释实施路径”“说明风险控制”
- 在 `slides/*.json` 中优先体现决策结构，而不是资料罗列
- 对总体方案，默认采用“总览页 + 分项设计页 + 关键机制页”
- 对分专题方案，默认采用“问题界定 + 设计思路 + 方案结构 + 落地方式”
- 对实施内容，默认采用“阶段路径 + 里程碑 / 依赖 + 责任分工”
- 对保障内容，默认采用“风险识别 + 应对机制 + 治理与组织”
- 若一章内存在多个独立子方案，应为每个子方案提供独立讲解页，而不是并在一张总表里带过
- 若输入未限制页数，不要为了“显得短”而跳过收益、风险、保障页

## `/fppt:detail` 细化要求
- `/fppt:detail` 的前提是项目目录内已存在并确认 `outline.json`
- `deck.json + slides/*.json` 的页序、章节组织和页面意图，必须与已确认的 `outline.json` 对齐；如需改结构，应先回到大纲阶段

- 背景与目标页：补充业务背景、建设动因、目标口径
- 现状与问题页：补充现象、影响、根因、证据
- 总体方案页：补充设计原则、方案框架、能力模块、边界说明
- 实施路径页：补充分阶段动作、里程碑、依赖项、责任角色
- 成本收益页：补充投入项、收益项、测算前提、时间范围
- 分项设计页：补充子方案目标、能力点、输入输出、与总体方案的关系
- 组织保障页：补充治理机制、职责分工、决策机制、推进要求
- 细化时若发现信息明显过载，应拆成多页，不要把关键模块继续压成列表

## 质检环节

方案类生成时，必须默认插入质检环节，而不是生成完就直接结束。

### A. `outline.json` 质检

在 `/fppt:outline` 产出 `outline.json` 后，必须先检查：

- 是否覆盖“背景 / 问题 / 方案 / 落地 / 收益 / 风险”完整链路
- 总体方案、分项方案、实施路径、风险保障是否已经拆开，而不是压在一页
- 每页是否标明明确决策作用，而不是资料罗列
- 是否存在多个子方案却没有独立讲解页的情况
- 是否存在明显页数过少，导致方案闭环不完整的情况

若以上任一项不满足，必须先回改 `outline.json`，不能直接进入 `/fppt:detail`。

### B. `deck.json + slides/*.json` 质检

在 `/fppt:detail` 生成详细页后，必须再检查：

- 页序、章节组织和页面意图是否仍与已确认的 `outline.json` 对齐
- 是否回答了“为什么做、做什么、怎么做、何时做、谁来做、有什么风险”
- 是否仍有页面只写结论，没有论据、路径或约束支撑
- 是否仍有“问题 + 方案 + 计划 + 风险”被硬塞在一页里的情况
- 是否把方案写成培训材料、产品介绍或泛泛而谈的建议清单
- 收益、风险、治理和落地要求是否已经讲清，而不是只停留在方案概念层

若以上任一项不满足，必须先回改对应页面，不允许直接交付。

### C. 质检输出要求

每次质检都应按固定模板输出检查结论：

1. `通过项`
2. `未通过项`
3. `修改动作`
4. `阶段判定`

推荐格式：

```md
质检结论

- 通过项：
  - ...
- 未通过项：
  - ...
- 修改动作：
  - ...
- 阶段判定：允许进入下一阶段 / 必须回改后再检查
```

其中 `阶段判定` 不能省略；如果没有明确写出阶段判定，则默认视为当前阶段未通过。

质检示例：

```md
质检结论

- 通过项：
  - 已覆盖背景、问题、总体方案、实施路径四个核心部分
  - 总体方案已经从现状分析中独立出来
- 未通过项：
  - 风险与保障仍未单独成页，当前闭环不完整
  - 两个子方案仍放在同一页里，缺少各自的目标、能力点和落地方式
- 修改动作：
  - 补一页“风险识别 + 应对机制”，再补一页“治理与组织保障”
  - 将两个子方案拆成各自独立页面，分别补齐目标、能力点、输入输出和实施关系
- 阶段判定：必须回改后再检查
```

## 推荐布局

- 背景/目标：`title_bullets`、`two_column`
- 现状/问题：`problem_statement`、`matrix_2x2`
- 总体方案：`architecture_layered`、`pyramid`、`logic_tree`
- 方案对比：`comparison_table`、`three_column`
- 实施路径：`roadmap`、`milestones`、`plan_table`
- 收益与风险：`cost_benefit`、`risk_register`、`raci`

## 可视化补充要求

- 方案类默认不能只靠文字论证，至少要显式产出“总体架构图 / 方案结构图”“实施路径图”“收益或风险页”三类视觉页面
- 总体方案页优先使用 `architecture_layered`、`logic_tree`、`svg_full`，不要只把模块名称堆成 bullets
- 实施路径页优先使用 `roadmap`、`milestones`、`gantt_chart`、`swimlane_process`，不要只写阶段说明段落
- 收益、优先级、资源投入和风险治理，优先使用 `cost_benefit`、`impact_effort`、`risk_register`、`comparison_table`
- 若没有现成示意图，也应在 `work/assets/` 下主动生成 SVG 结构图或流程图，不能把“建议放架构图”作为最终交付

## 文案风格

- 结论先行，但必须有论据支撑
- 关注“可落地、可执行、可评估”
- 多用“建议、应当、优先级、阶段、交付物、依赖、收益、风险”等方案语言
- 避免写成培训材料或产品宣传页

## 输出检查

- 是否回答了“为什么做、做什么、怎么做、何时做、谁来做、有什么风险”
- 是否具备从立项讨论进入执行准备的基本完整度
- 是否每个核心建议都能在后文找到支撑
