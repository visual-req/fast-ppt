## 适用场景

适用于产品演示、功能演示、解决方案演示、路演展示、对外介绍、能力展示等内容。

## 基本设计思路

演示类 PPT 的核心目标是“吸引注意、建立兴趣、强化记忆、促成认可”。整体设计应突出故事线、亮点和节奏，而不是展开完整论证。

默认采用“故事线演示模式”。这意味着：

- 默认先设计开场、亮点、转折、收尾的展示节奏
- 默认一页只突出一个记忆点，不把多个卖点压在一起
- 默认优先让观众“看懂、记住、想继续了解”，而不是看大量说明文字
- 只要一个能力点需要场景、过程、效果三部分支撑，就应主动拆成多页

推荐主线：

1. 受众痛点或场景引入
2. 核心卖点
3. 关键能力展示
4. 使用流程或体验路径
5. 效果对比
6. 应用场景或客户价值
7. 收尾与行动引导

## 章节模板

一个完整演示章节通常按以下顺序组织：

1. 场景引入：先把观众带入问题或机会
2. 亮点主张：这一段最想让观众记住什么
3. 能力展示：产品 / 方案具体能做什么
4. 使用流程：用户如何感知这个能力
5. 效果对比：相比旧方式好在哪里
6. 行动引导：希望观众下一步做什么

页数原则：

- 开场页和核心亮点页通常应独立成页
- 一个重点能力通常不少于 `2` 页：能力说明页 + 场景 / 效果页
- 若包含完整流程，通常需要“流程总览 + 关键步骤展开”
- 如果一页同时承担“场景、卖点、流程、效果”，通常说明还需要拆页

## `/fppt:outline` 生成要求

- `/fppt:outline` 必须先写出 `work/ppt/001_项目名/outline.json`，用于确认故事线、页序、布局选择和每页展示意图
- `outline.json` 是进入 `/fppt:detail` 前的必要检查点；未生成或未确认时，不允许直接生成详细页面内容
- 先明确受众是谁、最想打动对方什么
- 大纲要有明显的展示节奏：开场吸引、能力展开、亮点强化、收尾转化
- 每页都应服务于“展示一个亮点”或“推进一个场景”
- 复杂信息拆成多页分步呈现，避免一页塞满
- 如果包含功能流程，要按用户视角组织，而不是按内部模块组织
- 在 `outline.json` 中标明每页展示作用，例如“开场抓注意力”“说明场景痛点”“展示核心能力”“强化价值对比”“引导下一步”
- 对核心亮点，默认采用“亮点主张页 + 能力说明页 + 场景 / 效果页”
- 对流程展示，默认采用“总览页 + 关键步骤页”，不要把全流程塞在一张图上
- 对应用场景，默认采用“场景描述 + 用户动作 + 价值结果”结构
- 若主题中有 3 个以上核心卖点，应分别讲清，再做统一归纳，不要直接堆成一页卖点墙

## `/fppt:detail` 细化要求

- `/fppt:detail` 的前提是项目目录内已存在并确认 `outline.json`
- `deck.json + slides/*.json` 的页序、章节组织和页面意图，必须与已确认的 `outline.json` 对齐；如需改结构，应先回到大纲阶段
- 场景引入页：补充痛点、使用情境、目标用户
- 卖点页：补充一句话价值主张、差异点、关键词
- 能力展示页：补充功能点、亮点说明、使用收益
- 流程页：补充步骤、关键触点、转场说明
- 对比页：补充前后变化、传统方式 vs 新方式、价值提升
- 收尾页：补充行动建议、试用方式、下一步合作方向
- 亮点强化页：补充为什么值得记住、和竞品 / 旧方式的区别、适合谁
- 场景价值页：补充使用前状态、使用后状态、价值落点
- 细化时若发现一页文字解释过重，应优先拆页或改成图文对应，而不是继续堆说明

## 质检环节

演示类生成时，必须默认插入质检环节，而不是生成完就直接结束。

### A. `outline.json` 质检

在 `/fppt:outline` 产出 `outline.json` 后，必须先检查：

- 是否已经形成清晰故事线，而不是功能点散页堆叠
- 开场、亮点、能力、流程、效果、行动引导是否都已占到合适位置
- 核心亮点是否至少规划了“亮点页 + 能力页 + 场景 / 效果页”
- 流程内容是否已拆成“总览 + 关键步骤”，而不是整段塞在一页
- 是否存在 3 个以上卖点被直接堆在同一页的情况

若以上任一项不满足，必须先回改 `outline.json`，不能直接进入 `/fppt:detail`。

### B. `deck.json + slides/*.json` 质检

在 `/fppt:detail` 生成详细页后，必须再检查：

- 页序、章节组织和页面意图是否仍与已确认的 `outline.json` 对齐
- 是否有清晰开场、明显亮点、自然转场和明确收尾
- 是否每页只突出一个记忆点，而不是把多个卖点堆在一起
- 是否仍有页面文字过重、画面感不足、像方案说明而不像演示页面
- 场景页、流程页、效果页是否已经形成“看得懂 -> 记得住 -> 想继续了解”的推进节奏
- 收尾页是否已经明确行动引导，而不是泛泛结束

若以上任一项不满足，必须先回改对应页面，不允许直接交付。

### C. 质检输出要求

每次质检都应按固定模板输出检查结论：

1. `通过项`
2. `未通过项`
3. `修改动作`
4. `阶段判定`

推荐格式：

```md
质检结论

- 通过项：
  - ...
- 未通过项：
  - ...
- 修改动作：
  - ...
- 阶段判定：允许进入下一阶段 / 必须回改后再检查
```

其中 `阶段判定` 不能省略；如果没有明确写出阶段判定，则默认视为当前阶段未通过。

质检示例：

```md
质检结论

- 通过项：
  - 已形成“场景引入 -> 亮点 -> 能力 -> 收尾”的基本故事线
  - 核心卖点已经从开场页中独立出来
- 未通过项：
  - 三个卖点仍被堆在同一页，缺少各自的场景或效果页
  - 流程内容只有一张总览图，没有关键步骤展开
  - 收尾页还没有明确行动引导
- 修改动作：
  - 为三个卖点分别补“亮点页 + 场景 / 效果页”
  - 在流程总览后补 1-2 页关键步骤说明，突出用户动作和价值结果
  - 重写收尾页，补充试用、联系或下一步合作的明确引导
- 阶段判定：必须回改后再检查
```

## 推荐布局

- 开场/亮点：`cover`、`section_divider`、`title_bullets`
- 场景/旅程：`journey_map`、`steps`、`swimlane_process`
- 能力展示：`two_column`、`four_grid`、`before_after`
- 价值对比：`before_after`、`matrix_2x2`
- 收尾转化：`summary`、`thank_you`

## 可视化补充要求

- 演示类默认应以“图、流程、对比、场景感”为主，不能生成成一组汇报式文字页
- 每个核心卖点至少应配 `1` 张场景页、流程页、效果对比页或结构示意页
- 若没有真实产品截图，应主动在 `work/assets/` 下生成线框图、场景图、流程图或白板式 SVG 示意图，不能只写“这里建议放截图”
- 对使用流程优先使用 `journey_map`、`steps`、`swimlane_process`；对亮点差异优先使用 `before_after`、`matrix_2x2`；对能力结构优先使用 `four_grid`、`svg_full`
- 详细阶段要尽量形成“看图即可理解”的页面，而不是依赖讲者额外补充大量口头解释

## 文案风格

- 少字、多画面感、强动词
- 多用“亮点、价值、体验、场景、效果、提升”等展示语言
- 每页只突出一个记忆点
- 避免写成详细方案或经营汇报

## 输出检查

- 是否有清晰的故事线和展示节奏
- 是否能让观众快速记住 3 个核心亮点
- 是否避免复杂表格和过重文字负担
