本文件用于指导生成 `work/ppt/001_项目名/slides/*.json` 时如何选择 `layout_type`。

## 总原则

1. 每页只表达 1 个核心结论。
2. 先选表达方式，再填字段；字段必须符合 `layouts.md`。
3. 能用表清楚表达就用表；能用图更清晰表达就用图；没有数据就用占位符写清楚维度与口径。
4. 复杂内容拆页，不要挤在一页。

## 内容类型 → layout_type

封面与导航：

- 对外汇报开场：`cover`
- 章节/目录：`agenda`
- 章节分隔与转场：`section_divider`
- 阶段小结：`summary`
- Q&A：`thank_you`
- 补充材料：`appendix`

结论与论据：

- 单页结论 + 3-6 条要点：`title_bullets`
- 观点/证据对照：`two_column`
- 三方案并列对比：`three_column`
- 问题陈述（现象/影响/根因/证据）：`problem_statement`

结构化内容：

- 上下结构（上：结论/下：解释或证据）：`top_bottom`
- 步骤流程（按顺序推进）：`steps`
- 阶段路线（按阶段推进）：`phases` 或 `roadmap`
- 四宫格要点（4 个并列卡片）：`four_grid`
- 九宫格要点（9 个并列卡片）：`nine_grid`
- 象限分析（有两条维度的四象限）：`matrix_2x2`
- SWOT：`swot`

表格：

- 多方案/多维度对比：`comparison_table`
- 工作包计划：`plan_table`
- 风险台账：`risk_register`
- 里程碑清单：`milestones`
- 成本收益：`cost_benefit`

数据图表：

- 占比/构成（合计=整体）：`pie_chart` 或 `donut_chart`
- 已有高质量结构图/架构图/流程大图，需要整页展示：`svg_full`
- 类别对比（横向比较大小）：`bar_chart`
- 趋势变化（随时间/序列变化）：`line_chart` 或 `area_chart`
- 多维能力对比：`radar_chart`
- 计划排期（任务起止）：`gantt_chart`
- 结构分解/发散收敛：`mind_map`
- 转化漏斗：`funnel_chart`
- 变化分解（累计增减）：`waterfall_chart`
- 相关性与分布：`scatter_plot` / `bubble_chart`
- 强度分布矩阵：`heatmap`
- 层级占比：`treemap`
- 流向关系：`sankey`
- 单指标达成与目标：`gauge`

## 布局 → 场景映射（每种布局至少 1 个场景）

封面与导航：

- `cover`：对外汇报第一页（客户/日期/版本齐全），用于建立“主题 + 会议目的”。
- `agenda`：需要让听众预期结构（3-6 个章节），或需要对齐“本次汇报要解决什么问题”。
- `section_divider`：章节切换（例如从“现状与痛点”切到“总体方案”），用于把节奏拉回到主线。
- `summary`：阶段性结论收敛（例如“现状评估结论 4 条 + 下一步 2 条”），用于推动决策。
- `thank_you`：汇报结束进入 Q&A，或用于明确“下一步约定/联系人/会后材料”。
- `appendix`：需要把详细表格/定义/证据放到末尾，避免主线页过密。

文本与结构化表达：

- `title_bullets`：一个结论 + 3-6 条支撑要点（最常用通用页）。
- `quote`：需要引用客户原话/政策条文/会议纪要关键句，强化“需求真实性/合规依据”。
- `problem_statement`：需要把问题讲完整（现象→影响→根因假设→证据），用于立项与紧迫性建立。
- `two_column`：需要“左结论/右证据”，或“一侧要点 + 一侧表/图”，用于同时讲观点与数据。
- `three_column`：需要并列展示 3 个选项/3 条路径/3 类对象，用于方案选型或策略分层。
- `before_after`：需要明确对比“现状 vs 目标/方案后”（流程/组织/系统/指标），用于把变化点讲清楚。

结构化布局：

- `top_bottom`：需要“上结论/下解释或证据”或“上现状/下建议”，用于控制信息层级。
- `steps`：需要按顺序描述方法论/流程/推进步骤（例如“调研→分析→设计→落地”）。
- `phases`：需要按阶段描述推进（例如“试点→推广→规模化运营”），强调阶段目标与产出。
- `roadmap`：需要按时间/里程碑展示路线（可以是 bullets 或 table 占位），强调节奏与交付。
- `timeline`：需要按时间轴表达事件/里程碑（例如按月计划、关键会议、验收节点）。
- `four_grid`：需要 4 个并列要点（例如“4 类痛点/4 类能力/4 类收益/4 类风险”）。
- `nine_grid`：需要 9 个并列要点（例如“能力地图 9 宫格/功能模块 9 宫格/指标体系 9 宫格”）。
- `matrix_2x2`：需要 2 条维度下的象限分类（例如“价值×难度”“收益×风险”）。
- `swot`：需要从优势/劣势/机会/威胁四方面组织结论（适合战略与现状评估页）。
- `pyramid`：需要把“总论点→分论点→事实依据”一页讲清楚（执行摘要与核心结论常用）。
- `logic_tree`：需要输出问题树/假设树/指标分解树，并显式体现“验证路径/证据需求”。
- `icicle_tree`：需要展示层级分解（目标→子目标→举措，或能力→子能力→功能），强调路径与层级。
- `architecture_layered`：需要一页总览架构分层（业务/应用/数据/基础设施/安全/治理）与职责边界。
- `dependency_graph`：需要说明系统/工作包/数据之间依赖（计划、风险、关键路径说明）。
- `journey_map`：需要按旅程阶段对齐触点/痛点/机会点（体验改进与服务设计常用）。
- `swimlane_process`：需要跨部门/跨系统流程泳道图式表达（现状流程与改造流程对比常用）。

表格与清单：

- `comparison_table`：需要多方案、多维度对比（例如 A/B/C 方案在成本/周期/风险/收益上的对比）。
- `plan_table`：需要把工作包计划落到“工作包/负责人/周期/输出”，用于实施计划与项目管理。
- `milestones`：需要列出里程碑与验收点（例如 M1-M5、每个验收物与责任人）。
- `risk_register`：需要正式风险台账（风险/概率/影响/应对/触发条件/责任人）。
- `cost_benefit`：需要把投入与收益口径对齐（一次性/持续性成本、可量化/不可量化收益）。
- `raci`：需要明确职责边界（R/A/C/I），用于组织治理、流程责权、跨部门协同机制落地。

数据图表（有数据时优先，没数据时写清楚占位维度与口径）：

- `pie_chart`：需要表达构成占比（例如“费用结构”“问题类型占比”）。
- `svg_full`：已经有现成 SVG 大图，且图片本身就是主表达对象（例如架构总图、系统关系图、方法论全景图）。
- `donut_chart`：同 pie_chart，但希望中心显示“合计/关键指标”（可作为视觉焦点）。
- `bar_chart`：需要比较类别大小（例如“各部门需求量”“各系统缺陷数量”）。
- `line_chart`：需要表达趋势（例如“月活趋势”“故障率趋势”“工单量趋势”）。
- `area_chart`：需要表达趋势且强调“规模/累计”（例如“累积交付量”“累计成本”）。
- `radar_chart`：需要比较多维能力（例如“流程成熟度六维雷达”“能力差距对比”）。
- `gantt_chart`：需要表达任务起止与并行关系（例如“项目排期甘特图”）。
- `mind_map`：需要表达层级分解与发散（例如“问题树”“目标树”“能力分解”）。
- `funnel_chart`：需要表达转化漏斗（例如“线索→商机→签约→交付”）。
- `waterfall_chart`：需要表达变化分解（例如“成本从 A 到 B 的增减构成”）。
- `scatter_plot`：需要表达相关性（例如“投入 vs 收益”“风险概率 vs 影响”）。
- `bubble_chart`：需要 3 维表达（例如“价值/难度/规模”）。
- `heatmap`：需要表达矩阵强度（例如“部门×流程痛点强度”“系统×风险等级”）。
- `treemap`：需要表达层级占比（例如“预算树/成本树/模块占比”）。
- `sankey`：需要表达流向（例如“数据流/资金流/流程流转路径”）。
- `gauge`：需要表达单 KPI 达成（例如“可用性 99.9% vs 目标 99.95%”）。
- `impact_effort`：需要对“事项清单”做影响×投入的优先级排序，输出 quick wins 与战略投入项。

## 优先级与冲突处理

1. 有明确数据口径与单位 → 优先图表类。
2. 需要严谨、便于审阅 → 优先表格类。
3. 需要讲故事、便于汇报 → 优先结构化布局（top_bottom / steps / phases）。
4. 同一页出现“文本+表/图” → 用 `two_column`，将表/图放一侧，结论放另一侧。

## 视觉配额与兜底

1. 每个章节至少包含 `1` 张结构图 / 关系图页，以及 `1` 张图表 / 表格 / 图文页。
2. 不允许连续 `3` 页以上都使用纯文字表达；若前两页都偏文字，第三页应强制切换到图形化布局。
3. 对“演进、流程、架构、组件协同、规则闭环”类内容，优先图形布局，不要默认退回 `title_bullets`。
4. 若 `two_column` 的一侧是 `image`，详细阶段必须提供真实图片或 SVG 文件路径。
5. 当用户未提供素材时，应优先生成 `svg_full` 或改用可渲染的图表 / 结构图布局，不能只留下“建议配图”。
